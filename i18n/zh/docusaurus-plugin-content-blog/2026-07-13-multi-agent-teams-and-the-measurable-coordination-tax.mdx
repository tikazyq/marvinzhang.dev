设想一个十六人的工程团队。账面上，它是当初那个两人小组的八倍，预算也是照八倍给的。但如果把这个团队套进"组织到底怎么扩张"的数学里——公式我马上就摆出来——会掉出一个很别扭的数字：这十六个人，每周的产出大约相当于 **3.4** 个全职工程师。不是十六，不是八，是三点四。更糟的是，模型说这个十六人团队每周的产出，比它自己在八个人时还要 **少**：产能在八人左右见顶，然后掉头往下。剩下那十二个多人的工资并没有闲着，他们很忙——忙着协调、忙着等待、忙着把一件事重新解释一遍、忙着把两个人对不上的理解拉齐。这些活是真的活，只是跟你当初招人时想要的活，不是一回事。

先说清楚：上面这些数字来自一个**示意模型**，不是真去查了哪家公司的工资单——是一条我待会儿会把旋钮都交给你看的拟合曲线，不是对真实公司的调查。具体数值别抠得太死，要死死盯住的是那条曲线的 **形状**，因为形状才是全部的故事，而它刚刚在一个新地方冒了出来。

把人换成 AI 编码 agent，曲线并不会因为感恩就变平。2025 年底，一个来自 MIT 和 Google Research 的团队做了[一项关于 agent 系统扩展的对照研究](https://research.google/blog/towards-a-science-of-scaling-agent-systems-when-and-why-agent-systems-work/)，覆盖 180 种配置。在真正能拆开并行的任务上，多加 agent 帮助很大——中心化协调比单个 agent 高出 **80.9%**。但在必须一步接一步的顺序任务上，他们测过的每一种多 agent 方案都比单个 agent 更差，差了 39% 到 70%。在单个 agent 的推理能力被摊得太薄之前，他们的实用上限是三到四个 agent。另一个 2026 年的基准 [Silo-Bench](https://arxiv.org/abs/2603.01045) 发现，仅仅到两个 agent，你就已经丢掉了单个 agent 能做到的 **15% 到 49%**。

同一条曲线。当年让你十六人团队只干出三个人产出的那个病，如今正一分一毫地在你的 agent 集群里复发。而值得写一篇文章的地方在于：这个病已经五十岁了，早就诊断得清清楚楚——但在组织的历史上第一次，我们终于能把它**精确地量出来**。

*两种模式都听命于同一个方程。过了最佳点，你每多加一个单位，带来的产出都比上一个少——直到不再带来产出，再往后就是倒扣。那个下坡不是你团队的毛病，它是协调这件事本身的性质。*

## 给这个病起名字的那道数学

这条曲线有名有主。2007 年，Neil Gunther 在《Guerrilla Capacity Planning》里提出了 **[通用可扩展性定律](https://www.perfdynamics.com/Manifesto/USLscalability.pdf)（Universal Scalability Law，USL）**，本来是想解释为什么给数据库加 CPU 加到后来就不管用了。结果它能描述任何"一群各干各的工人共用某样东西"的系统。公式在这里，只出现这一次——之后我保证把它收起来：

```
              λN
X(N) = ───────────────────────
        1 + σ(N−1) + κN(N−1)
```

三个旋钮，每一个都能翻译成关于团队的大白话：

- **λ——单兵产能。** 一个工人自己干、没人可聊时能干出多少。分子里的 `λN` 是幻想：完美的线性扩张，N 个人干出 N 倍的活。线下面的一切，都是被抽走的税。
- **σ——抢占成本。** *抢那件共用东西*的代价——那一把数据库锁、那一套联调环境、那一个所有人都要找的评审。它随 N 线性增长。烦，但扛得住。
- **κ——对齐成本。** *让所有人保持在同一页上*的代价。这个才是要命的，因为它带着 `N(N−1)` 这一项——它随团队人数的 **平方**增长。每进来一个新人，都得跟已有的每一个人对齐。

那个 κ·N² 项不是什么新想法。它是把 Fred Brooks 的[《人月神话》](https://en.wikipedia.org/wiki/The_Mythical_Man-Month)（1975）写成了代数。Brooks 数了 n 个人的团队里有多少条沟通路径，得到 n(n−1)/2：四个人有六条两两之间的通道，十六个人就有一百二十条。增长是平方级的，这也正是为什么"往一个已经延期的软件项目里加人，只会让它更晚交付"。Gunther 的 κ，就是给每一条路径贴上的价签。

代数还顺手把那条著名的建议白送给了你。对 X(N) 求导、令其为零，峰值就落在 `N* = √((1−σ)/κ)`。按 widget 里的人类参数，这个值大约是八。所谓"两个披萨能喂饱的团队"从来不是江湖传说，它是一条曲线的闭式最大值；而且它会往你预料的方向移动：把 κ 调大，N* 就往左滑，滑向更小的团队。

要给这两个系数配个画面，就用厨房。σ 是几个厨子挤在一起抢那一个灶台——很闹心，但闹到头也就那样。κ 是所有人在挤满人的厨房里扯着嗓子喊菜、生怕谁把菜出错——而这喊叫声随着招人，是平方级地越来越吵。人类发明的解法是小票制度：一笔结构性投资，让厨子们通过一套便宜的流程来协调，而不是同时进行一百二十场对话。记住这个小票制度，文章最后会绕回到它。

公式到此为止，你不会再见到它了。接下来的问题不是*这条定律是什么*——Gunther 早就把它定死了——而是*当工人是由 token 做成的，我们终于能拿它做点什么*。

## 主体一：agent 团队的病历本

带着 USL 去读那些 agent 研究，它们就不再像一堆意外的清单了，倒像一个你早就见过的团队。

先看失败。在 [《为什么多 agent 大模型系统会失败？》](https://arxiv.org/abs/2503.13657)——也就是 MAST（多 agent 系统失败分类）背后那篇、来自伯克利、在 NeurIPS 2025 上发表的论文——作者们手工标注了七个主流框架上的一千六百多条执行轨迹。失败率并不好看：从大约 40% 一路到 86%，其中 ChatDev 做对的时候少到只有 25%。但真正有意思的是诊断。把失败归归类，大约 **37%** 属于他们所说的*agent 间对不齐*——agent 各说各话、丢上下文、把队友早就定过的决定又重新吵一遍。另有 42% 是需求界定的问题，21% 是验证没跟上。用 USL 的话说，中间这三分之一就是被逐条列出来的 κ，就是"在厨房里扯嗓子喊"那一桶，而且是最大的一桶协调开销。

MIT–Google 那篇给同一个直觉配上了系数。在他们的各种配置里，判断一个多 agent 系统到底是帮忙还是添乱，最强的单一预测因子是他们命名的**工具–协调权衡**（tool-coordination trade-off，β = −0.330）：一个任务越依赖用工具，协调开销就越会把好处吃掉。他们还量了错误怎么传播。让 agent 各自为战、没有协调者，一个错误在无人拦截地扩散时会被放大 **17.2 倍**；把一切都走一个中央协调者，这个放大就被压到 4.4 倍。

仔细看这笔交易，因为它是管理学里最老的一招。编排器（orchestrator）就是 agent 世界的项目经理。派一个人管事，能把 κ 压下去——不再是每个 agent 都要跟其他每个 agent 对齐，而是都通过协调者对齐，那一堆平方级的路径就朝着一颗星星塌缩。但协调者得碰到每一件事，而一个碰到每件事的东西，就是一个串行的瓶颈。你压低 κ，是靠**让自己变成了 σ**。用平方级的协调成本，换一条线性的排队，正是人类项目经理做的那笔买卖，也正是为什么这个解法确实管用、却从来不白给。

现在讲讲另一面，因为故事不是"多 agent 就是坏"。当任务本身就能并行，多加 agent 会赢，而且赢得漂亮。Anthropic 报告说，他们的[多 agent 研究系统](https://www.anthropic.com/engineering/multi-agent-research-system)在广度优先的研究任务上，比单个 Claude Opus 4 高出 **90.2%**——这种活你要的就是十个各自散开的图书管理员，而不是一个人从头读到尾。Google 自己的数据也同意：在能并行的任务上，协调正好*帮了忙*，帮的就是那 80.9%。Cognition 的工程团队先是在[《别搭多 agent》](https://cognition.com/blog/dont-build-multi-agents)里论证，脆弱的上下文共享让多数多 agent 方案都是个错误——几个月后又在[《真正跑得通的多 agent》](https://cognition.com/blog/multi-agents-working)里讲了这个模式在哪能真正顶用：读操作散开，写操作留在单线。两篇文章指向的，是 MIT–Google 那队直接说出来的同一条规律——**是架构和任务对不对得上，决定了协作能不能成**。并行的活奖励更多 agent，顺序的活惩罚更多 agent。这不是悖论，这就是那条曲线：κ 高的任务，它的 N* 被死死摁在一附近，再聪明的编排也挪不动一个平方律的峰。

## 主体二：第一个真能被测量的组织

到这里为止的一切都是老戏重演。接下来才是真正的新东西——它讲的不是 agent 比我们差，而是 agent 以一种任何人类组织都不曾有过的方式，变得**可读**了。

经济学家想测量协调成本，想了一个世纪。他们基本测不了，因为面前挡着三堵墙。

第一，**没有日志。** 你没法给一家公司装探针，没法给走廊里的一次交谈打上性能剖析，没法记下两个人脑子里的理解悄悄分岔的那一刻，也没法把一场例会里烧掉的"token"加起来。人类协调的原始事件流，压根就没被记录，从来没有。

第二，**你没法重跑。** 想干干净净测出团队规模的代价，最好的办法是拿同一个季度，用八个人跑一遍、再用十六个人跑一遍，其他一切都摁住不动，然后把产出一减。没有哪家公司能这么干。历史只跑一次。所有关于"团队多大才合适"的说法，都是套了件白大褂的个人经验，因为那个反事实根本没法观测。

第三，**人会应付指标。** 你一旦测量某样东西、还挂上后果，[古德哈特定律](https://en.wikipedia.org/wiki/Goodhart%27s_law)（Goodhart's Law）就来咬你：这个指标就不再测得准了。你数代码行数，就会得到又臭又长的代码；你数关掉的工单，工单就会被拆碎。人会去应付一切他们看得见的观测手段，于是把你本想信任的那个数字给腐蚀了。

agent 一下子把这三堵墙全推倒了。

**日志是完整的。** agent 的每一次交互，本来就是走 API 的 token，也就是说它本来就被记了账、打了时间戳、齐全无缺。没有哪次走廊交谈能逃过账本，因为根本没有走廊。组织经济学从没有过的那条事件流，对一个 agent 系统来说是 **默认就在那儿的产物**——你得费劲才能把它扔掉。

**N 是一个自由变量。** 拿一批固定的任务，用两个 agent 跑，再用四个、八个、十六个，其他全钉死。这就是没有哪家公司买得到的那个反事实，而对 agent 来说，它的代价是一个下午加一点推理预算。Silo-Bench 整套方法干的正是这件事——扫过不同的 agent 数量、摁住任务集、看会发生什么。你正在跑那个人类组织设计只能做白日梦的随机对照实验。

**而且 agent 不会应付账本。** 一个 agent 不会为了升职而虚报自己的 token 数，它根本不知道有这套账在记。腐蚀人类指标的那个古德哈特效应，在这儿压根没有落脚点。数字之所以老实，是因为系统里没有谁有动机去掰弯它。

于是我们终于能测量人月了。但是——这一步很容易做错——*测得好，并不等于"把 agent 之间的消息数一数"*。那个直觉是个陷阱，两个近期的结果正好把它踩响。

第一个是 [AgentTaxo](https://openreview.net/forum?id=0iLbiYYIpC)，它剖开看多 agent 框架里的 token 到底花在哪。协调成本大头并不在那些看得见的交接消息上，它藏在冗余的共享上下文里、藏在验证里——在 MetaGPT 的一个实验里，光验证这一个阶段就吃掉了绝大部分 token。协调里最贵的，是安静的那部分：同一份上下文，被反复重发、反复核对。你要是只数 agent 之间的对话，就会漏掉这张账单的大半。

第二个更彻底。[LatentMAS](https://arxiv.org/abs/2511.20639) 让 agent 完全在**隐空间**里协作——传的是隐藏状态的向量和一份共享的隐式记忆，而不是自然语言文本——而且这么干还少用了 70% 到 84% 的输出 token。再读一遍：可能 **根本没有消息可数**。协调发生了，只是它从没变成过文字。任何等价于"把 agent 间的消息加起来"的测量方案，碰到一个正在你没法 token 化的表示里疯狂协调的系统，读数都会是零。

那么，老实测量协调成本的办法，跑在两条轨道上，而且两条都不要求你把责任摊到每一项上。

**第一条轨道是整体测量。** 什么都别去拆。改变 N，把产出画出来，看那条曲线怎么从理想的 `λN` 直线上弯开去。那道缝就是协调税——你是靠这个病的总体效果去测它，而不是去解剖到底哪个器官衰竭了。Silo-Bench 把这一点浓缩成一个干净的数字，**RCC = 1 − SR(k)/SR(1)**：你从单 agent 走到 k 个 agent，损失掉的那部分成功率。MIT–Google 的扩展定律是同一个想法拟合成的曲线。这一步底下压着一个世纪的科学哲学——[迪昂–奎因](https://en.wikipedia.org/wiki/Duhem%E2%80%93Quine_thesis)（Duhem–Quine）论题：你能观测到一个系统整体上偏离预测多少，却没法把这偏离赖到里面任何一条具体假设头上。协调成本正是这一类量：在总体上干干净净可测，在细节上确实没法拆成"这条消息花了我们 3%"。

**第二条轨道是一组过程指标**——留给你想去调方向、而不只是想打个分的时候。像 AgentTaxo 那样的上下文重复率记账。agent 利用率：每个 agent 的工作里，有多少是真在干活、有多少是在等。每次委派的成本。错误放大率，也就是那个 17.2 倍对 4.4 倍的落差。这里每一个都是一个代理指标——一个对 κ 某个侧面的、局部而老实的观察——没有哪个是*那个*协调成本。把它们当一块仪表盘看、而不是当一句判词，它们能告诉你该拧哪个旋钮。

重点不是 agent 难协调。人类更难。重点是 Brooks 和 Gunther 几十年前就把正确答案递给了我们，而我们从来没法*验算他们的作业*——组织死活不肯让人装上探针。agent 是第一个说"行"的组织。人月从一句好记的格言，变成了一次测量。

## 接下来变稀缺的是什么

退得够远，这些转变里有个反复出现的规律。每一波自动化，都会把从前是瓶颈的那样东西变成大路货，而稀缺、值钱的本事，就跳到别处去。

AI 正在把*干活*变成大路货——就叫它 W，那些原始的活：写好的函数、跑通的测试、立起来的接口。agent 把这些批发掉了。当执行变便宜，约束就挪到执行必须保持一致的那一切上头，而这个约束的名字，我们整篇文章都在绕着它转。它就是 κ。

接下来这几年稀缺的本事，不是更快地写更多代码，而是在不把自己变成那个瓶颈的前提下、把 κ 压下去：设计 agent 借以协调的协议，划好边界让活能并行而不是撞车，定出机器真能核对的验证标准，搭出行为像小票制度、而不是像喊菜大战的编排。这是工程，有一个可测量的目标函数——协调税——而且第一次，仪表盘上有一根实时的指针把它读给你听。

上个时代，活是提高 λ——让每一个工人，不管是人还是机器，单个更能干。这局基本赢了，agent 够快了。这个时代，活是成为那个把 κ 压低的人。曲线一直都在，我们只是终于能实时看着它弯下去了——而且，终于能对这个弯，做点什么了。
