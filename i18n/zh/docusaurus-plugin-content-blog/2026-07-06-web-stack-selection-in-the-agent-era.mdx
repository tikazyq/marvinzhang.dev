2025 年 3 月 11 日，Anders Hejlsberg 发表了一篇题为 [A 10x Faster TypeScript](https://devblogs.microsoft.com/typescript/typescript-native-port/) 的公告：微软要把 TypeScript 编译器移植到 Go。

对大多数读者来说，这是一条性能新闻。对我来说，这是三篇旧文的迎头相撞。2021 年，我在短短几个月里先后写过三篇彼此不相干的文章：一篇问 [大红大紫的 Golang 真的是后端开发中的万能药吗](/blog/golang)，一篇论证 [为什么 TypeScript 是开发大型前端项目的必备语言](/blog/typescript)，还有一篇夸 [C# 的开发体验](/blog/csharp)——那篇文章里我顺带提过一句当时看来只算趣闻的事实：TypeScript 和 C# 出自同一人之手。五年后，三篇文章的主角在同一个仓库里会师：我称为"必备"的那门语言的编译器，用我当年审视过的那门语言重写，主导者正是 TS 与 C# 共同的父亲。我知道这几篇文章之间有联系，但没想到会以这种方式收束。

这场会师的价值不止于怀旧，它暴露了争论底下悄悄变掉的东西。2021 年，咱们争的是哪种语言对"人"更友好：谁的语法更干净，谁的学习曲线更平缓，谁的类型系统更不烦人。这场争论至今没有结束，但裁判换了。从那之后，AI 编程智能体（coding agent）成了生产代码最高频的作者之一，也是代码的第一读者。agent 不在乎 Go 的错误处理有多啰嗦，也不嫌 TypeScript 的类型标注有多累赘。它只关心两件事：一是 **多快拿到反馈**，二是 **反馈能不能被信任**。

这个观察就是本文的候选论点，我刻意用它的克制版本来陈述：当 agent 加入你代码库的作者行列，web 技术选型多出两个升为一等公民的判据——**agent 循环延迟**（一次"修改代码 → 获得可信反馈"的端到端耗时）和 **可判定验证信号密度**（每一轮循环中，能被机器自动、确定性地判定的正确性信号有多少）。这两个判据不会取代以人为本的旧判据，但会给它们重新排序。而在新的排序之下，2021 年那场语言之争的判决，有的被加冕，有的被改判。

为了把这个论证做得诚实，接下来我会先做一轮站内考古，看看 2021 年我们究竟在争什么；然后给两个新判据下精确定义；再检视 2025 到 2026 年的三组证据；随后让最强的反方论证单独占一节；最后给出一个小小的选型框架——你完全可以不同意它，但至少可以有效率地不同意。

## 2021 年，我们在争什么

重读自己五年前的技术文章，是一种奇特的考古体验。器物都还眼熟，真正让人吃惊的是埋在器物底下的那些假设。

我的 [TypeScript 篇](/blog/typescript)（2021 年 2 月）翻来覆去只讲了一个论点：类型是协作契约。JavaScript 的自由度是全文的反派——"动态一时爽，重构火葬场"，我当年引用的这句坊间流言说得够直白了。我最推崇的机制是 **错误前置**：违反契约的代码在编译时就以一条确定性的报错浮出水面，而不是等到生产环境里变成一句 `undefined is not a function`。不过当时我把这一切完全框定为人类团队的收益：接口即文档，自动补全即舒适。

我的 [前端工程化篇](/blog/frontend-engineering)（2021 年 3 月）事后看来，是一份打扮成课程表的抱怨。它罗列了一个"真正的"前端工程师必须掌握的东西——Webpack、Babel、Node、NPM/Yarn、ESLint、TypeScript 配置——并且承认"稍微大一点的前端项目的依赖文件就会超过 500MB"。文章结尾半开玩笑地问：今天入坑前端的你，还学得动么？在那篇文章里，工具链的复杂度被当成理所当然的行业现状：我只琢磨怎么掌握它，从没想过它本身就是问题。

我的 [Golang 篇](/blog/golang)（2021 年 3 月）拒绝给任何东西加冕。它的方法论是明确的辩证法："Golang 的一些突出特性将成为它的双刃剑"——语法简单限制了表达复杂问题的能力；强制错误处理让 `err` 满天飞；而编译速度，我在优缺点对照表里开了个玩笑，是唯一一行没有缺点的特性（"怎么可能是缺点？"）。那篇文章甚至做了一个可证伪的预测：泛型大概要等 Go 2.0。结果这个预测还保守了：[Go 1.18 在 2022 年 3 月就带来了泛型](https://go.dev/blog/go1.18)，连版本号都没进位。

我的 [C# 篇](/blog/csharp)（2021 年 11 月）引入了我如今看来最耐用的分析框架：满意度与市场份额的错位。引用 StackOverflow 2021 年对 82,914 名开发者的调查：市场巨人 Java 的用户满意度只有 47%，而它的"山寨版"C# 拿到了 62%。我当时的解读是：**满意度是领先指标，市场份额是滞后指标**。那篇文章的标题里写着"Part 1"，Part 2 至今没有出现。这个梗先记下，文末有用。

![四篇 2021 年文章并列：各自的判据、当年结论与 2026 年读来的成色——所有判据的量纲都是人](/img/blog/2026-07-06-web-stack-selection-in-the-agent-era/figure-1-2021-criteria.zh.png)

把四篇文章这样排成一行，一个 2021 年谁都没想过要说出口的共同假设就显形了：所有判据都以 **人** 为量纲。人的学习曲线，人的阅读舒适度，人的满意度问卷。没有人问过机器会如何体验这套技术栈——因为当时没有机器在写代码。

## 裁判换了

观察一个 agent 改完代码之后做什么。它跑类型检查，跑测试，读报错，然后再改。取决于工作流的设计，这个循环在一个任务里会闭合几十次、上百次——而一个人类工程师，一天闭合几十次就到头了。于是你的技术栈有两个属性开始支配 agent 的吞吐量，而这两个属性在我 2021 年的判据清单里一个都没有。

**循环延迟** 是一次循环的端到端耗时：安装、编译、类型检查、测试启动、拿到反馈。对人来说，70 秒的类型检查是个可以起身倒杯咖啡的烦恼；对一个要迭代几百次的 agent 来说，延迟就是吞吐量本身，它会乘进所有环节。我 2021 年 Golang 对照表里那个玩笑行——编译速度"怎么可能是缺点？"——就这样悄悄升格成了一等选型判据。

**可判定验证信号密度** 是每轮循环中机器无需人类判断就能裁决的正确性总量。一条 `tsc` 报错是零成本产出、确定性、机器可读的：同一个缺陷永远给出同一条消息，指向确切的文件与行号。生产环境里的一个运行时 `TypeError` 则处处相反：触达昂贵、复现随机、还需要人来解释。这个区别的分量今非昔比，因为 agent 是机械地消费这些信号的。一项 2025 年针对 LLM 生成代码的研究（[GitHub 工程博客引用](https://github.blog/ai-and-ml/llms/why-ai-is-pushing-developers-toward-typed-languages/)）发现：**LLM 产出代码的编译错误中，94% 是类型检查失败**——换句话说，从经验数据看，类型系统是 agent 所能遇到的密度最高的验证信号发生器。

![agent 循环：修改代码 → 构建/类型检查/测试 → 信号分叉——确定性、机器可读的信号让 agent 自我纠正并循环成百上千轮；微弱的仅运行时信号则迫使人类介入。技术栈决定循环延迟与信号密度。](/img/blog/2026-07-06-web-stack-selection-in-the-agent-era/figure-2-agent-loop.zh.png)

本文机制层面的核心，就压在这一句话上：**训练数据决定 agent 首轮写得多好，验证信号决定它多快收敛到对。** agent 工作流天生是多轮的，所以主导经济账的是收敛变量，而不是首稿变量。这句话是全文的承重墙，反方证词一节里我会专门对它做压力测试。

也该交代一下这个论点在既有讨论中的位置，因为它的零件并不全新。Daniel Demmel 的 [反馈循环工程（feedback loop engineering）](https://www.danieldemmel.me/blog/feedback-loop-engineering) 把循环质量讲得很透，但刻意保持工具无关——它拒绝谈选型。Netlify 提出的 [AX（agent experience）](https://biilmann.blog/articles/introducing-ax/) 点破了优化目标的迁移，但落点是"平台如何服务别人家的 agent"。我还没有见到有人把这两股线接回我 2021 年那几篇文章真正讨论的那个决策——**你到底选哪套技术栈**——并且直面一个两边都没有处理的约束：代码库既要让人可审，又要让 agent 可循环。这两个诉求是会打架的：显式、冗长利于 agent，却在消耗人类读者。"人机联合 DX"不是一句口号，而是一个文末框架必须兜得住的取舍。

## 判决正在被改写

三组来自 2025–2026 年的证据，每一组都在某篇 2021 年的旧文里有回声。

**证据一：typescript-go。** 2025 年 3 月宣布的移植，在 [2026 年 6 月进入 RC](https://devblogs.microsoft.com/typescript/announcing-typescript-7-0-rc/)（`typescript@rc`；截至本文写作，稳定版仍未发布，npm 的 `latest` 还停在 6.0.3）。微软自己在 2025 年 12 月的实测里给出 7.5–10.2 倍的提速区间——团队现在的措辞是"通常约 10 倍"（often about 10 times faster），比发布时的标题谨慎了一档。用本文的两个判据去读这个决策，它就不再只是一次性能优化：业界资金最充足的编译器团队之一，花数年把一个好端端的编译器移植到另一门语言，买的就是 **同样的信号密度、十分之一的延迟**。论点的两个变量，被一个真实的工程决策同时标了价。回声几乎工整得过分：TS 篇的"错误前置"论证，从人类的便利升格为机器的口粮；Golang 篇那个玩笑行，成了这笔采购的全部理由；而 Hejlsberg 弃自己的 C# 选 Go——[他给出的理由](https://github.com/microsoft/typescript-go/discussions/411) 是现有代码风格的可移植性，而非语言忠诚——本身就是"判据变了"的一次现场演示：连两门语言共同的父亲，如今也按循环力学来挑语言。

**证据二：Hono 对 Express。** [State of JS 2025 调查](https://2025.stateofjs.com/en-US/libraries/) 把 Hono 放进了 S 级——满意度超过 90%，与 Vite、Vitest 同档——而这只是它第一年进入调查；同一份报告写道，Express 的使用率仍是第一，"但满意度指标暴露了它的年纪"。如果这个结构让你觉得眼熟，那就是我 C# 篇里那张图换了名字：用户容忍的市场巨人，对上用户喜爱的后来者。满意度领先，市场份额滞后——2021 年的框架原封不动搬过来就能用。不过对本文论点而言，Hono 真正有意思的地方在于它究竟为什么令人满意。它的 [RPC 模式](https://hono.dev/docs/guides/rpc) 把服务端类型直接共享给客户端——不用代码生成，不经容易漂移的 OpenAPI 中间层；它的 Zod 校验集成也绝非边缘玩法：单是 `@hono/zod-validator` 在 npm 注册表官方接口口径下就有约 270 万次周下载。用选型的语言说：这是一个把"验证信号"论证做成 **全栈契约** 的框架——服务端改坏了 API 形状，客户端连编译都过不去。

```ts
// 服务端：路由 + 校验器共同定义契约
const app = new Hono().post('/users', zValidator('json', userSchema), handler)
export type AppType = typeof app

// 客户端：请求/响应形状由 tsc 检查——传错 body 是编译错误，不是 500
const client = hc<AppType>('http://localhost:3000/')
```

按本文的精神，数字要交代口径：Hono 的原始下载量（写作时注册表官方接口约 4700 万次/周）已经使它成为仅次于 Express 的第二大 JS 后端框架——但 2026 年整个 npm 注册表的数字都被 CI 与自动化流量大幅推高了，所以我只为相对结论背书：Hono 的增速显著跑赢这个被灌水的大盘，而它与 Express 的走势对比，就是"满意度对市场份额"的错位在实时上演。

**证据三：Bun，先听控方的一半。** Bun 的卖点是一个二进制吞下运行时、包管理器、打包器和测试运行器。它的安装速度是各路独立评测中复现最一致的优点；[State of JS 2025](https://www.infoq.com/news/2026/03/state-of-js-survey-2025/) 显示 21% 的受访者在使用它（同比 +4 个百分点）。这里的回声是我前端工程化篇的核心抱怨——500MB 的依赖目录、课程表一样的工具清单。我在 2021 年默默忍受的复杂度，事后被证明是一个市场：Bun 的整个叙事，就是把那条工具链归零。但我当年写 Golang 篇，恰恰是为了抵抗这种叙事——同样的辩证法在这里同样锋利，所以 Bun 的另一半证词，留给下一节的辩方。

![三组 2026 年证据与 2021 年回声的对应：typescript-go 加冕错误前置与编译速度论点；Hono 对 Express 复用满意度对市场份额框架；Hono RPC 让类型契约跨过网络边界；Bun 回应了工具链之怨——但双刃剑同样锋利](/img/blog/2026-07-06-web-stack-selection-in-the-agent-era/figure-3-verdicts.zh.png)

## 反方证词

一个过于顺手的论证，配得上最强的反对。三条反方证词，按威胁程度从高到低。

**训练数据飞轮。** [MultiPL-E 基准](https://nuprl.github.io/MultiPL-E/) 确立了一个事实：LLM 各语言的代码生成正确率，与该语言训练语料的体量强相关；社区讨论里也反复引用悬殊的对比——一个流传很广的基准称模型写 Elixir 的正确率接近 97%，写 Rust 只有 61%。GitHub 自家的 [Octoverse 2025](https://github.blog/news-insights/octoverse/typescript-python-and-the-ai-feedback-loop-changing-software-development/) 描述了由此形成的飞轮：越流行的语言生成质量越好，于是变得更流行。如果首稿质量才是决定性的，那么选型规则简化为"选最流行的"，我的两个判据不过是装饰。我的回应就是前文那句机制话：语料体量设定每轮循环的起点，验证密度设定的是 **收敛速率**，而多轮工作流对后者是复利。但我必须对证据现状保持诚实：截至写作，还没有任何一项控制变量的研究，能在隔离这两个变量的前提下度量 agent 跨技术栈的端到端任务成功率。本文的论点建立在机制与行为证据之上，而不是一个已经宣判的结论——这正是全文采用克制版本论点的原因。

**Bun 自己就是反例。** 把 2021 年 Golang 篇的方法——那张每个优点都投下阴影的表——原样用到 Bun 身上，阴影相当可观：

![用 2021 年 Golang 篇的方法审视 Bun——每个优点配上阴影：单二进制对兼容表面积与 5000+ 未关闭 issue；速度优先文化对内存泄漏与 OpenCode 迁出；AI 加速节奏对生产稳定性折价](/img/blog/2026-07-06-web-stack-selection-in-the-agent-era/figure-4-bun-double-edge.zh.png)

社区形成的务实共识——开发用 `bun install` 提速，生产继续跑 Node——本身就是一份判决书：**循环延迟的优化，替代不了运行时的可靠性。** 在文末的框架里，延迟只是一条轴，不是整个平面。不过 Bun 最有教益的一个数据点，恰恰是双向的。2026 年 6 月，[Prisma 把 Compute 产品迁到了 Bun 的 Rust 重写版 canary 上](https://www.prisma.io/blog/bun-rust-rewrite-prisma-compute)——因为他们实测发现 **稳定版** 内存泄漏、连接池死锁，而 canary 通过了他们的负载测试。他们的原话值得抄录："The build which passed our tests made a better foundation than the build which failed them."（通过了我们测试的构建，比没通过的那个更适合当地基。）版本标签是弱信号，跑通的测试套件才是可判定信号。连 Bun 的麻烦，最后都在给本文的论点作证。

**秤本身歪了。** 最后再交代一件诚实的事。语言之争里我们挥舞了多年的下载量——2021 年我自己也挥过——正在作为证据变质。2026 年上半年，npm 注册表的流量几乎全线翻倍（Express 的月下载量从约 2.2 亿涨到约 4.6 亿，找不到任何合理的人类解释），普遍归因于 CI 与自动化 agent 流量。我们用来度量"流行"的仪器，如今有相当一部分在度量机器。这里头有种冷幽默的公道：一个呼唤更硬验证信号的时代，恰好也是我们最软的那个信号——下载计数器——悄悄失去含义的时代。

## 一个两轴选型框架

如果这两个判据是真的，它们就应该能用。下面是它们张开的平面；各点的位置是基于前文证据的 **定性判断**，不是打分。

![两轴平面：循环延迟 × 验证信号密度。TS 7 原生版、Go、Hono + Zod 位于联合 DX 甜区；JS 版 tsc 的 TS 可信但迟缓；纯 JavaScript 与典型 Python 配置快而信号稀疏；Bun 现状最靠右但带可靠性折价。](/img/blog/2026-07-06-web-stack-selection-in-the-agent-era/figure-5-quadrant.zh.png)

比象限图更重要的，是使用它的三条规则。

**第一，这两条轴是入场券的补充，不是替代。** 运行时可靠性与生态成熟度依然是不可谈判的入场资格——Bun 在延迟轴上再激进，只要可靠性折价没有补齐，生产环境就不认。任何象限位置都压不过一个会死锁的连接池。

**第二，"联合"意味着人的判据保留投票权。** 我 2021 年 Golang 篇结尾的建议——评估候选技术与业务的切合度、与团队的融合度、以及学习与时间成本——原样成立；你只需要把这套评估跑两遍，一遍为你的人，一遍为你的 agent，然后把冲突摆到明面上谈。一个被优化成显式、冗长、agent 易读、但让资深工程师害怕评审的代码库，没有达成联合 DX——它只是把一种受众的痛苦换给了另一种受众。

**第三，语料充足是前提条件，不是坐标轴。** 这个框架只在训练数据饱和的主流生态 **内部** 做比较。选一门精致但冷门的语言，等于送给 agent 一份注定糟糕的首稿，多高的验证密度都补不齐——这正是训练数据反方论证的合理内核，在此吸收为边界条件。

这个框架拒绝生产一种句式："X 是 agent 时代最好的语言。"2021 年我拒绝为 Go 写这句话——"如果一门编程语言能广为人知，那它绝对不会是一门糟糕语言"——五年过去，我只会拒绝得更坚决。判据可以旅行，王冠不能。

## 当主体变成客体

这次研究里有一个进展，放在 2021 年会荒诞得像段子。Bun——我通篇作为 agent 的"选项"来掂量的那个工具——如今在很大程度上是由 agent"建造"的。Anthropic 于 2025 年 12 月收购该项目后，按 [RedMonk 的统计](https://redmonk.com/sogrady/2026/06/04/bun-two-lessons/)，其提交中由 bot 完成的比例升到了 80% 以上；2026 年 5 月，一次由编程智能体在九天内生成的、超过百万行的 Rust 重写，以单个合并落入主干。我要谨慎地引用它：这是一家有充分动机做展示的厂商的展示性事件，也只是关于一个代码库的孤例。但作为路标，它很难被替代。2021 年的问题——**哪种语言最适合人来写**——有了一个旧争论的语法里根本装不下的后继问题：哪套技术栈，最能让机器来写、让人来裁决、并让双方都信得过结果。

这个重新表述，正好补上考古一节留下的两个尾巴。前端工程化篇的结尾问：今天入坑前端的你，还学得动么？2026 年的答案出奇地温柔：你不必全学了——agent 替你消化了半张课程表。但有一层你无法外包：课程表之上的判断层——**由哪些判据来决定你的技术栈**，这件事你得自己想清楚，因为不管你递给 agent 什么样的循环，它都会兴高采烈地在里面优化。至于 C# 篇欠了五年的 Part 2——就当是这一篇吧。只是续集的主角，从一门语言，换成了给语言排座次的标准。
