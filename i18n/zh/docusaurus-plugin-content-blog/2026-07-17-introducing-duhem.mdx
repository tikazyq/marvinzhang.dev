AI agent 写代码很快。如今的瓶颈已经不是写，而是——在你不盯着、不手动点一遍的情况下——怎么知道它交付的东西是不是真的能跑通，端到端地跑通。

今天你只有两个都不太好的选择。

一是**每次改动后手动测一遍**。这跟不上 agent 的速度；更糟的是，它恰恰看不见最要命的那类问题：应用 **自己报告“健康”**，但某条真实路径已经悄悄坏了。你看一眼，好像没事，就发布了。

二是用 Playwright 之类的工具写**端到端（e2e）测试**。但 e2e 测试不是你写一次的规格说明，而是一段你要一直维护下去的 **程序**：异步、等待、重试、flaky、fixture……它会慢慢偏离“做完”最初的含义，而且 agent 一改底层实现，它就断。

我们做 **[Duhem](https://github.com/onsager-ai/duhem)**，是因为还有第三个选择——而且在别人需要之前，我们自己的产品先需要它。

## 它是什么

Duhem 是面向 AI 构建的软件的**整体性验证（holistic verification）**。你写下验收标准，它把标准转成可被机械判定的检查，让这些检查去运行你真正交付的东西——代码、提示词、工具、数据、运行时，全部一起——再用判定结果决定这次改动能否合并（merge）。四点让它不一样：

- **规格即测试。** 一个 Verification Definition 就是一个产物：你可读的意图，绑定着可运行的检查。你读的、跑的、卡住合并的是同一样东西——所以规格、测试、代码不会各自悄悄漂移。
- **不 mock。** 它驱动的是真实交付的产物、走真实的接口。绿灯意味着真实系统跑通了，而不是某个替身跑通了。
- **判定里没有 AI。** agent 可以帮你起草标准，但标准由人来负责；通过与否是确定性的。没有模型给自己打分，也就没有“自信但错误”的绿灯。
- **验证不会腐烂。** 你写的是“要验证什么”，而不是“验证它的那段程序”——锚定在意图上，而非实现上。agent 把内部重写一遍，检查的含义依然不变。

这是说辞。下面是真事。

## 那次抓到的

Crawlab Pro——我们的爬虫平台——在把 `:edge` 镜像晋级为 `:stable` 之前，会用一套 Duhem 套件给它把关。有一天这道关亮了红灯：*环境起不来。*

原因几乎是隐形的。一个新镜像发布时少了一个小工具 `envsubst`。它的缺失，让容器启动脚本悄悄渲染出了一份**空的** nginx 配置。nginx 找不到配置，退回到默认页——于是整个 Web 前端在所有人访问的那个端口上彻底挂了，`/api` 没了。

接下来这段，应该会让你有点不安。**后端是好的。** 而容器自带的健康检查——就是 Docker 和 Kubernetes 用来判断“这东西还活着吗”的那个——报告的是 **健康**，因为它查的是后端仍在写的一个健康*文件*，而不是真正坏掉的前门。一个人点进去，会看到页面加载失败，然后大概率耸耸肩。产品自带的每一个自动化信号都说：绿。

Duhem 说：红。前端不通，就不通过。它不在乎健康检查怎么说，它在乎的是真实系统有没有做真实的事。它拒绝给一个前端已死的构建盖章——而且是在这个构建能晋级 `:stable`、能触达任何一个用户**之前**拒绝的。

我们修好镜像，同一道关验证了修复，现在它在每次构建后自动运行。用户影响：零。

一次事故，就是全部价值所在。**Duhem 抓住了一个自我伪装的回归——而其他所有信号，包括产品自己的健康检查，都在说它健康。** 这正是手动测试看不见、常规测试通常够不着的那类 bug，也正是会一路溜到生产、让你付出代价的那类。

## 这是给谁的

Duhem 是给那些**业务关键、复杂、且持续变化**的软件的技术负责人的——在这些系统里，一次静默的回归代价高昂，而“它还能用吗”不是你扫一眼就能确认的事。

它**不**适合原型、一次性的小应用，或者任何“想要它的人自己看一眼就知道行不行”的场景。如果你靠看就能验证，你不需要 Duhem——比起把它卖给你，我们更愿意直接告诉你这一点。

## 和我们一起试

Duhem 是开源的，而且还很早期——v0.x，会有破坏性变更，我们会帮你迁移。我们在找一小批、运行着真实且业务关键的 AI 构建系统的人，来试用它、和我们一起把它打磨好：直接的支持，直接对接做这件事的人。
