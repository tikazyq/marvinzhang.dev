Six protocols. Six automation levels. Seventeen tools. Twelve predictions. One interactive map that ties them all together.

The [AI Agent Interaction Landscape](https://agent-landscape.vercel.app) is an open-source, bilingual SPA I built to make sense of how AI agents interact with developers, editors, tools, and each other in 2026. This article walks through the key frameworks it introduces—and the insights that emerged from building it.

## Why a Landscape Map?

The AI agent space in early 2026 feels like the JavaScript framework wars of 2016—except the stakes are higher and the acronyms multiply faster. MCP, ACP, A2A, A2UI, AG-UI, AP2: six protocols from four different organisations, each addressing a different layer of the agent interaction stack. Meanwhile, tools proliferate across CLIs, IDEs, mobile apps, and headless platforms, each supporting different subsets of these protocols at different automation levels.

The landscape map exists because no single article, talk, or documentation site captures the full picture. It's not a product comparison. It's a **protocol stack visualisation** that shows how these pieces fit together—from the human endpoint (your phone, terminal, IDE) down to the execution substrate (cloud containers, local runtimes, LLM routing).

## The Protocol Stack: Five Layers

The core of the landscape is a five-layer protocol stack. Each layer addresses a different boundary in the agent interaction model.

### Layer 1: Human Endpoints

Four surfaces where developers interact with agents:

| Endpoint | Mode | Key Pattern |
|----------|------|-------------|
| **Mobile** | Async control | Intent dispatch, approval, notification-driven |
| **Terminal** | Full development | Text streaming, pipe composition, persistent sessions |
| **IDE / Desktop** | Rich review | ACP access, generative UI, diff review |
| **Headless** | Unattended | CI/CD, webhooks, bots, IoT |

The insight here isn't that these are four alternatives—they're four **projections** of the same agent system. A developer might dispatch an intent from their phone on a train, have the agent execute via terminal on a remote machine, review the diff in their IDE at the office, and let headless agents handle CI overnight. Same system, different views.

### Layer 2: Agent-Human Protocols

Three protocols standardise how agents communicate with humans:

- **ACP** (Agent Client Protocol) by JetBrains + Zed — the "LSP for AI agents." A standard interface between editors and AI agents, making the "which editor supports which agent" question irrelevant.
- **A2UI** by Google — declarative UI specification that lets agents produce native renders on Web, Flutter, SwiftUI, and Compose.
- **AG-UI** by CopilotKit — event-driven bidirectional sync between agents and frontends, handling state synchronisation and tool call events.

Plus a **Generative UI** pattern that maps the control-freedom tradeoff across three modes: static (AG-UI), declarative (A2UI), and open-ended (MCP Apps).

### Layer 3: The Agent Harness

This is where the engineering happens—four subsystems that make agents production-ready:

- **Governance**: `evaluate(action, ctx, state) → Allow | Deny | Modify`. Interceptor patterns, rule crystallisation, budget/rate/safety boundaries.
- **Runtime**: Agent loop state machines, session management, recovery. Plan → Approve → Execute → Review cycles with token budgeting and exponential backoff.
- **Capability**: Tool / Skill / Agent hierarchy with typed schemas (Pydantic/Zod). Skill evolver repair and hot-reload.
- **Coordination**: Factory, fractal, and generative adversarial patterns for multi-agent orchestration.

As automation increases from L2 to L5, the harness weight grows: none → Capability → +Governance → +Runtime → +Coordination. At L4-L5, the harness **is** the product—it's the only thing between autonomous agents and production.

### Layer 4: Agent-Agent/Tool Protocols

Four protocols for machine-to-machine communication:

| Protocol | Owner | Purpose |
|----------|-------|---------|
| **MCP** | Anthropic | Agent↔Tool — "USB-C for AI." JSON-RPC servers/clients |
| **A2A** | Google / Linux Foundation | Agent↔Agent — delegation, discovery, mesh |
| **Agent Spec** | Oracle | Framework-agnostic agent definition |
| **AP2** | Google | Agent payment — cart mandates, scoped contracts |

MCP is the most mature (1.0 shipped), but A2A is catching up fast with AgentCard metadata and cross-organisation mesh support.

### Layer 5: Execution Substrate

Where agents actually run:

- **Cloud**: Per-agent isolated containers (Codespaces, DevBox, Sealos). Scale-to-zero.
- **Local**: CLI local with direct file/Git access (Claude Code, Codex, Gemini CLI).
- **Hybrid**: On-device + cloud offload, following the Apple Intelligence pattern.
- **LLM**: Model routing and token budgeting. Haiku → Sonnet → Opus tiering.

## The L0→L5 Automation Spectrum

Perhaps the most useful framework in the landscape is the six-level automation spectrum, modelled after autonomous driving levels:

| Level | Name | Human/Agent | Key Shift |
|-------|------|-------------|-----------|
| **L0** | Manual | 100% / 0% | Memory-driven |
| **L1** | Assisted | 80% / 20% | Searching → generating answers |
| **L2** | Collaborative | 55% / 45% | Writing code → reviewing changes |
| **L3** | Semi-Autonomous | 25% / 75% | Writing code → describing intent |
| **L4** | Fully Automatic | 5% / 95% | Approving steps → setting boundaries |
| **L5** | Orchestration | 2% / 98% | Defining workflows → defining goals |

The industry mainstream sits at **L2** (collaborative), with the frontier pushing into **L3** (semi-autonomous). The jump from L2 to L3 is where the real paradigm shift happens: the developer stops writing code and starts describing intent. The jump from L3 to L4 is even more radical—the developer stops being in the loop at all, and sets governance boundaries instead.

### The Same Bug, Six Ways

To make the abstraction concrete, the landscape shows the same bug fix at every level:

- **L0** (4 hours): Read Jira, search StackOverflow, trace code manually, write fix, wait for review.
- **L1** (2 hours): Paste error to ChatGPT, copy-paste the suggested fix, test manually.
- **L2** (45 minutes): Describe bug in Cursor chat, AI scans codebase, proposes inline fix, human approves with one click.
- **L3** (10 min human + 15 min agent): Type one sentence in terminal, agent plans/executes/tests/commits, human approves PR on phone.
- **L4** (0 min human + 8 min agent): CI detects failure, headless agent diagnoses and fixes, governance auto-approves, auto-deploys to staging.
- **L5** (30 min goal-setting + 6 hours agent mesh): Human defines migration goal, architect agent designs plan, coding agents implement in parallel, security agent audits.

The pattern is clear: human time drops exponentially, but the **weight of each remaining human decision increases**. A 30-second approval at L4 can greenlight 8 hours of agent work.

## A Day in the Life at L2-L4

The landscape includes a "Day in Life" view showing how these automation levels weave together across a real workday:

- **07:30** (Phone, L4): Approve overnight refactoring results with a swipe. 28 files changed, 412 tests passing. 15 seconds.
- **08:15** (Phone, L3): Dispatch a bug fix from Slack via Claude Code Remote Control while ordering coffee.
- **09:30** (IDE, L2): Pair-program OAuth integration with Cursor. Developer architects, AI implements. 2 hours, ~30% human-written.
- **11:45** (Phone, L3): Bug fix from morning complete. Approve from phone, PR auto-merges.
- **14:00** (IDE, L2): AI-assisted code review catches missing index, N+1 query risk.
- **16:00** (Terminal, L4): Configure overnight agents—dependency upgrade + security audit. 5 minutes of setup, 8+ hours of autonomous work.
- **22:00** (Phone, L4): Quick dashboard check. Agent A at 45% progress. Agent B completed, filed 2 tickets. No alerts.

Total: **~3.5 hours** of human time, **~11 hours** of agent time, across 3 devices and 3 automation levels. The phone isn't a coding tool—it's an **approval surface**.

## The Tool Ecosystem: 17 Products Mapped

The landscape maps 17 tools across their protocol support and automation range:

**CLI Agents** dominate the L2-L5 range:
- **Claude Code** (Anthropic) — L2-L5, MCP + ACP. 1M token context. The current frontier leader.
- **Gemini CLI** (Google) — L2-L4, MCP + ACP. Plan Mode and Conductor architecture.
- **Codex CLI** (OpenAI) — L2-L4, MCP + ACP.
- **Aider** (Open Source) — L2-L3, MCP. AI pair programming.
- **OpenCode** (Open Source) — L2-L3, MCP + ACP. ACP-compatible.

**IDEs** cluster at L1-L3:
- Cursor, Windsurf, JetBrains AI, Zed, GitHub Copilot. All now support MCP; JetBrains and Zed lead with native ACP.

**Mobile** targets L3-L4:
- Claude Code Remote Control, Pocket Agent, Moshi. The phone as async control surface.

**Frameworks** reach L5:
- CopilotKit (AG-UI creator), Google ADK (A2A + A2UI), LangGraph (stateful orchestration).

**Platforms** at L4-L5:
- BeeAI by IBM/Linux Foundation. Open-source agent orchestration.

The key observation: **protocol support determines automation ceiling**. Tools with only MCP cap out around L3. Adding ACP enables IDE integration. Adding A2A + AG-UI unlocks L4-L5 multi-agent mesh.

## Eight Insights

The landscape's "WHY" section distils eight core observations:

1. **"TUI revival isn't nostalgia"** — AI output is natively a text stream. The terminal is the most efficient text stream renderer ever built.

2. **"MUI won't happen"** — There won't be a mobile UI standard for agents. A2UI already makes mobile a native render target.

3. **"Your phone is an approval surface, not a coding tool"** — The bottleneck on mobile isn't input (AI handles that)—it's output. Phones are optimal for approve/reject decisions.

4. **"Four endpoints are projections, not alternatives"** — Mobile, Terminal, IDE, and Headless aren't competing. They're different views of the same agent system.

5. **"Less time, more leverage"** — From L0 to L5, human time drops from 100% to 2%, but impact per decision increases exponentially.

6. **"Governance is the real product"** — At L4/L5, governance is the only thing between agents and production. The most valuable AI infrastructure in 2027 won't be the smartest model—it'll be the best guardrails.

7. **"Protocols > Products"** — MCP, ACP, and A2A will outlive today's AI tools, just like HTTP outlived Netscape.

8. **"The future of coding is async"** — At L4+: set intent → agent works overnight → review in the morning.

## Security: Six Threats, Six Defences

As agents push into L4-L5, security becomes the critical bottleneck. The landscape identifies six threat vectors:

1. **Over-permissioned agents** — An L4 agent with unrestricted file system access deletes production config during a refactoring task. **Defence**: Least-privilege per task, allowlists over denylists, time-limited permission envelopes.

2. **Broken trust chains** — Agent A delegates to Agent B via A2A, inadvertently granting broader permissions. **Defence**: Capability downscoping at each delegation hop, permission decay with each handoff.

3. **Prompt injection propagation** — Malicious instructions in code comments hijack agent behaviour, spreading through multi-agent systems. **Defence**: Sanitise at every boundary, canary tokens, cross-validate multi-agent outputs.

4. **Unintended data exfiltration** — An agent sends secrets to external logging during debugging. **Defence**: Data classification labels on MCP resources, network segmentation, token-level redaction.

5. **Governance bypass via tool composition** — Individual tools are safe, but composing them creates dangerous capabilities. **Defence**: Analyse action sequences, estimate blast radius, require human approval above thresholds.

6. **Approval fatigue** — After approving 50 routine requests, a developer rubber-stamps a security vulnerability. **Defence**: Risk-based routing, attention signals for high-impact changes, randomised attention checks.

The most insidious is #6. Every other threat has a technical defence. Approval fatigue is a **human factor** problem that erodes the entire human-in-the-loop safety model.

## The Time-Leverage Paradox

The deepest insight from building this landscape is what I call the **time-leverage paradox**: as human time approaches zero, the value of each remaining human moment approaches infinity.

At L0, you spend 4 hours on a bug fix. Every minute is roughly equal. At L4, you spend 0 minutes—the agent handles it. But when something does require your attention, that 30-second approval decision greenlights 8 hours of autonomous work. At L5, a 30-minute goal-setting session produces days of agent mesh output.

This isn't just a productivity story. It's a fundamental shift in what it means to be a developer. The scarce resource isn't coding time anymore—it's **judgment**. The ability to set the right boundaries, approve the right plans, and catch the right edge cases. Governance is the real product because it's the codification of that judgment.
