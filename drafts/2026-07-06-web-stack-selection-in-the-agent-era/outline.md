**核心命题(弱形式,B 骨架的纲)**

> 当 AI agent 成为代码的主要作者之一,web 技术选型多出两个一等公民判据:**agent 循环延迟**(一次"改代码→获得可信反馈"的耗时)与**可判定验证信号密度**(单位循环内机器可自动判定的正确性信号量)。这两个判据重新排序了 2021 年那场语言之争的胜负——当年为"人类 DX"吵出的结论,有的被加冕,有的被改判。

## § 3. 裁判换了 — 两个新变量(700–850 words)

**Beats**

- 场景切入(C 皮):一个 agent 改完代码后发生什么——它跑 tsc、跑测试、读报错、再改。它不抱怨语法啰嗦,不嫌错误处理无聊;它只关心两件事:**多快拿到反馈**、**反馈可不可信**。
- 操作化定义两个变量(命题正式陈述,弱形式,明确"是判据升格,不是判据替换"):
  - **循环延迟**:安装、编译、类型检查、测试启动的端到端耗时。人类一天几十次循环,勉强忍;agent 一小时几百次,延迟即吞吐。
  - **可判定验证信号密度**:tsc 报错 = 零成本、确定性、机器可读;运行时 undefined is not a function = 昂贵、随机、需要复现。94% 的 LLM 生成编译错误是类型检查失败(GitHub Blog 引学术研究)——类型系统恰好是信号密度最高的发生器。
  - 关键机制句(可引用级):训练数据决定 agent **首轮写得多好**,验证信号决定它**多快收敛到对**;agent 工作流是多轮的,循环变量才是主导项。
- 与既有话语的差异化(点名但克制):Demmel 的 "feedback loop engineering" 讲了循环、拒谈选型;Netlify 的 AX 讲了目标迁移、停在平台层;本文把两者接到**选型决策**上,并用"联合 DX"框定张力——人要可审,agent 要可循环,两者会打架(显式冗长利 agent、累人眼),后文框架须同时容纳两者。
- **Visual**:Mermaid 流程图(agent loop,标注延迟点与信号点)。代码片段候选 1:并列两行报错(编译时 vs 运行时)——写作时若表格/文字足够则删。

## § 4. 改判进行时 — 三组证据(850–1000 words)

> 每组证据都带"2021 旧文回声",这是 C 皮的主要着力点。

**Beats**

- **证据组 1:typescript-go(判据一与判据二在同一工程决策中交汇)**
  - 事实链(全部用 research.md 已核验口径):2025-03 公告 → 2026-06-18 RC 进主线包 → 截至写作日稳定版未发(npm latest 6.0.3);性能 "often about 10x"(12 月实测 7.5–10.2x,写谨慎口径)。
  - 微软肯花这个成本,买的就是"同样的信号密度,十分之一的延迟"。
  - 回声:TS 篇"编译时错误前置"论证升维;Golang 篇"编译迅速:这绝对是一个优点/怎么可能是缺点?"的玩笑升格为核心判据;Hejlsberg 弃 C# 选 Go 的理由本身就是一次"判据变了"的现场演示(port 可行性 > 语言忠诚)。
- **证据组 2:Hono vs Express(C# 篇框架原样复用)**
  - State of JS 2025:Hono 满意度 >90% 进 S-tier、后端满意度前三;Express usage 第一"但满意度指标反映出它的年龄"。与 2021 年 Java/C# 错位同构——**满意度是领先指标,市场份额是滞后指标**,这句 2021 年的判断今天再用一次。
  - 为什么是 Hono 而不是别的:Hono RPC + Zod(@hono/zod-validator 2.72M/周,官方接口口径)= 全栈类型契约,前后端共享同一套可判定信号;Web Standards 多运行时 = 循环里少一层适配噪音。
  - 下载量谨慎呈现:官方接口 46.8M/周 vs 第三方旧快照 1.8M/周,口径差 20 倍+,正文只写"注册表口径下已是 Express 之后第二、且增速显著高于大盘",细节脚注化。
  - 代码片段候选 2:Hono RPC 类型共享 3–5 行——若 §3 已用片段则此处改文字描述。
- **证据组 3:Bun(只取循环延迟侧,短板留给 §5)**
  - 单二进制吃掉 runtime + 包管理 + 打包 + 测试;安装提速是多源交叉最一致的优点;State of JS 2025 使用率 21%(+4pt)。
  - 回声:前端工程化篇的"500MB 依赖"与工具清单之怨——当年的复杂度诉求,今天以"归零"的形式被回应。但立刻埋钩子:"归零"的代价,下一节算账。

## § 5. 反方证词 — 训练数据、双刃剑与失真的秤(750–900 words)

> 本文可信度的支点。三个反方论证按强度排,不稀释。

**Beats**

- **反方 1:训练数据飞轮(最强反方)**。MultiPL-E:LLM 各语言正确率与语料量强相关;HN 高赞引 Elixir 97% vs Rust 61%。如果 agent 写得好主要靠语料,选型标准就是"选流行的"。回应(即 §3 机制句的展开):首轮质量 vs 收敛速度,两者相乘而非互斥;并诚实声明:目前**没有**控变量的端到端实证,本文命题是基于机制与行为证据的推断,不是已证定理。
- **反方 2:Bun 自己就是反例(Golang 篇方法论原样复用)**。用 2021 Golang 篇同款优缺点对照表审视 Bun:极简单二进制 ↔ Node 兼容长尾(官方自认 16 部分兼容 + 3 未实现);快迭代 ↔ 生产稳定性(长跑内存泄漏 issue、OpenCode 迁出、5k+ open issues、"开发用 bun 生产跑 node"的社区共识)。结论句:**循环延迟优化不能替代运行时可靠性**——两轴框架里它只占一轴。Prisma 案例在此出场(canary 修好稳定版泄漏),但按双面证据呈现。
- **反方 3:失真的秤(度量失真 meta 观察,短)**。2026 年 npm 大盘因 CI/agent 流量近乎翻倍,"下载量"这个 2021 年式论据本身失去可判定性——评估工具的信号也在经历同一场信号革命。一段收尾,带一点讽刺,不展开。
- **Visual**:Bun 优/缺点表,**表头与行文格式刻意复刻 2021 Golang 篇的对照表**(致敬即论证:方法论五年后仍然好用)。

## § 6. 一个两轴选型框架(600–700 words)

**Beats**

- 立轴:X = agent 循环延迟(低→高),Y = 可判定验证信号密度(低→高)。右上象限 = 联合 DX 最优区。
- 摆点(基于前文已核验证据,不引入新数字;明示这是**定性判断**,不是打分):TS(信号密)、TS+tsgo(信号密且延迟骤降)、Go(双高但生态位不同)、Python(语料王者、信号弱——训练数据派的堡垒)、Bun(延迟轴激进、可靠性折价)、Hono+Zod(框架层把信号密度做进契约)。
- 使用规则(比象限图更重要,三条):
  1. 两轴之外,**运行时可靠性与生态成熟度仍是入场券**,不是可交易项(Bun 教训);
  2. "联合"意味着人侧判据不清零——可审性、团队熟悉度仍在(2021 Golang 篇的选型三问:业务切合度、团队融合度、成本——照抄仍成立,只是要为 agent 再问一遍);
  3. 训练数据充足是前提项:框架在主流生态内比较才有意义,冷门语言先输在首轮质量。
- 收束:这不是"XXX 是 agent 时代最好的语言"——2021 年我拒绝写这种句子,2026 年也一样(Golang 篇原话回收:"如果一门编程语言能广为人知,那它绝对不会是一门糟糕语言")。

## § 7. 结尾 — 主客体塌缩与一个旧问题的新答案(350–450 words)

**Beats**

- 主客体塌缩(线索⑤,克制呈现):2021 年我们争"哪种语言适合人写";写这篇文章时,Bun 仓库 80%+ 的 commits 由 bot 提交,百万行重写由 agent 完成——被选型的工具正在被 agent 建造。标注幸存者偏差与展示性事件属性,不神化。
