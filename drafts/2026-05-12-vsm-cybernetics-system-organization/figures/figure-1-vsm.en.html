Figure 01 — Viable System Model

Five levels, one side channel, one alarm path

Beer's Viable System Model rendered against a software organisation. Every level lives somewhere; the question is where. On the left, the role that has historically carried it. On the right, the equivalent in a pipeline driven by AI coding agents.
Algedonic (alarm) path
S3* — side channel
Subsystem cells
Algedonic · S1 → S5, bypassing S2/S3/S4
S5
Policy
Identity, purpose — the unfalsifiable commitments.
In a traditional team
Founder statements, company values, exec all-hands talking points.
In an agent pipeline
AGENTS.md, CLAUDE.md, principles docs, architectural decision records.
S4
Intelligence
Outward and forward — what should be on the backlog.
In a traditional team
Product, user research, sales intel, customer success, competitive analysis.
In an agent pipeline
Almost always the part that doesn't exist. No organ for noticing what should enter the backlog.
S3
Control
Here-and-now scheduling, allocation, supervision.
In a traditional team
Scrum master, engineering manager — standups, sprint planning, capacity allocation.
In an agent pipeline
DAG planner, work-allocation policy, session triage — what most teams call "agent tooling".
side channel — parallel, not subordinate
S3*
Audit
Traditional: SRE, QA, security review — outside the engineering reporting chain.
Agent pipeline: audit of routing/planner/supervisor, independent of the operators.
S2
Coordination
Keeps S1 units from clobbering each other.
In a traditional team
Branch policies, PR review queue, paired-programming protocols, the unspoken rules.
In an agent pipeline
Git worktrees, sandboxed runtimes, request-scoped DB fixtures, locks around shared writes.
S1
Operations
The executing unit — the part that actually does the work.
In a traditional team
The developer writing code, the test running, the deploy executing.
In an agent pipeline
A single agent session — one coding-agent invocation that ships a pull request.
Source: Beer, Brain of the Firm (1972) · Diagnosing the System for Organizations (1985)
