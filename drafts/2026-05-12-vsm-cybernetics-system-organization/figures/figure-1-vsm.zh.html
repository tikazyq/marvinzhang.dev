Figure 01 — 可生存系统模型

五个层级，一条侧通道，一条警报通路

比尔的可生存系统模型映射到一个软件组织。每一层都活在某处——问题只是活在哪里。 左边是它在传统工程团队里历史上的角色，右边是在跑 AI 编程智能体的流水线里的对应。
痛快通道（警报路径）
S3* — 侧通道
子系统单元
痛快通道 · S1 → S5，绕过 S2/S3/S4
S5
政策
身份与目的——不可证伪的承诺。
在传统团队中
创始人发言、公司价值观、高管 All-Hands 反复强调的几个要点。
在智能体流水线中
AGENTS.md、CLAUDE.md、原则文档、架构决策记录（ADR）。
S4
情报
朝外、朝前——什么本来应该出现在 backlog 上。
在传统团队中
产品、用户研究、销售情报、客户成功、竞品分析。
在智能体流水线中
几乎总是不存在的那部分。没有任何器官在留意什么应进入 backlog。
S3
控制
此时此地的调度、分配与监督。
在传统团队中
Scrum Master、工程经理——站会、迭代规划、产能分配、阻塞解除。
在智能体流水线中
DAG 计划器、工作分派策略、会话分诊——多数团队口中的"智能体工具"。
侧通道 — 平行，不是下属
S3*
审计
传统场景：SRE、QA、安全审计——在工程汇报链之外。
智能体流水线：对路由系统/计划器/监督逻辑的审计，独立于运行它们的人。
S2
协调
防止 S1 单元互相踩踏。
在传统团队中
分支策略、PR 评审队列、结对编程协议，以及不成文的规则。
在智能体流水线中
Git worktree、沙盒化运行时、按请求作用域的 DB fixture、共享写路径上的锁。
S1
作业
执行单元——真正在干活的那部分。
在传统团队中
写代码的开发者、跑测试的进程、执行部署的脚本。
在智能体流水线中
一次智能体会话——一次 coding agent 调用，交付一个合并请求。
来源：比尔，《企业之脑》（1972）·《组织系统诊断》（1985）
